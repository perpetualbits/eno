### 1.2 Setting Up the Self-Hosted RustDesk Relay
To keep your presentation and session authentication entirely private, spin up a dedicated coordinator on the Proxmox host via Docker Compose:

# /opt/rustdesk-server/docker-compose.yml
version: '3'
services:

## 2. Simultaneous Mixed-OS Virtualization Pathways

To run Linux and Windows VMs concurrently on the same machine without license servers, select one of the following hypervisor strategies:

### APPROACH A: Zero-Cost Hardware vGPU Partitioning (vGPU_Unlock)
* **Best For:** Splitting the physical VRAM of a single GPU (like the RTX A5000) into true hardware slices, allowing Windows and Linux guests to run 3D workloads simultaneously.
* **Licensing Fee:** \$0 (Bypasses NVIDIA Virtual Workstation grid software checks).

#### 1. Prepare the Proxmox Host Core
Isolate the host graphics card configurations by blocking standard drivers upon startup:
```bash
echo "blacklist nouveau" | sudo tee -a /etc/modprobe.d/pve-blacklist.conf
echo "blacklist nvidia" | sudo tee -a /etc/modprobe.d/pve-blacklist.conf
```
Clone and build the community driver override package to trick the kernel into accepting vGPU profiles on standard hardware layouts:
```bash
git clone https://github.com
cd vgpu_unlock && ./install.sh
```

#### 2. Create the Simultaneous VM Seats
Use Proxmox's hardware configuration manager to assign distinct, partitioned Mediated Devices (`mdev`) to each guest.

*   **Seat 1: Windows 11 Guest (Claiming an 8GB Slice):**
    ```ini
    # /etc/pve/qemu-server/101.conf
    name: windows-vgpu-seat
    ostype: win11
    cores: 8
    memory: 16384
    balloon: 0
    machine: pc-q35-8.1
    hostpci0: mapping=NVIDIA-A5000-vGPU,mdev=nvidia-225,pcie=1
    vga: none
    ```
    *Note: Inside the Windows guest, install the standard NVIDIA Enterprise/Grid desktop driver. It will expose full, native DirectX 11/12, CUDA cores for Ollama, and NVENC video encoding.*

*   **Seat 2: Linux Ubuntu Guest (Claiming an 8GB Slice):**
    ```ini
    # /etc/pve/qemu-server/102.conf
    name: linux-vgpu-seat
    ostype: l26
    cores: 4
    memory: 8192
    balloon: 0
    machine: pc-q35-8.1
    hostpci0: mapping=NVIDIA-A5000-vGPU,mdev=nvidia-225,pcie=1
    vga: none
    ```

### APPROACH B: Dedicated Core Splitting & Network API Bridging
* **Best For:** Environments that require maximum stability without using experimental host overrides.
* **Mechanism:** Dedicate the full hardware card to the Windows guest, and configure the Linux guest to offload its AI workloads across the network.

#### 1. Proxmox Configuration for Fat Windows Seat (Direct Hardware Access)
Pass the physical PCIe boundary directly to the Windows guest:
```ini
# /etc/pve/qemu-server/103.conf
name: fat-windows-seat
ostype: win11
cores: 8
memory: 16384
balloon: 0
machine: pc-q35-8.1
hostpci0: 0000:03:00,pcie=1,x-vga=1
vga: none
```

#### 2. Proxmox Configuration for Headless Linux Seat (CPU/Network Rendered)
```ini
# /etc/pve/qemu-server/104.conf
name: cross-api-linux-seat
ostype: l26
cores: 4
memory: 8192
machine: pc-q35-8.1
vga: virtio
audio0: device=ich9,driver=spice
```

#### 3. Redirect Linux AI Computation Over the Network Mesh
Inside the Linux VM, skip downloading heavy model weight layers. Open your user environment profile:
```bash
nano ~/.bashrc
```
Add the environment variable telling the system to transparently route all local command prompt AI matrices straight to the Windows instance via Tailscale:
```text
export OLLAMA_HOST="100.X.Y.Z:11434"
```
*(Replace `100.X.Y.Z` with your Windows VM's private Tailscale IP string).*

## 3. Guest Workspace Automation Engine

Configure these automation loops inside the guest operating systems to handle multi-monitor scaling, audio forwarding, and remote USB redirection.

### 3.1 Display & Sunshine Streaming Automation
Install the high-performance Sunshine runtime layer inside the guest:
```bash
sudo add-apt-repository ppa:loki-va/sunshine -y && sudo apt update
sudo apt install sunshine xserver-xorg-video-dummy -y
sudo rustdesk --option allow-linux-headless Y
```

To enable low-overhead, synchronized audio captures, configure a PipeWire virtual loopback device to route guest sound down to your client device:
```bash
pactl load-module module-null-sink sink_name=sunshine_sink sink_properties=device.description=SunshineAudio
```
*In the Sunshine Web UI (`https://localhost:47990`) $\rightarrow$ **Configuration** $\rightarrow$ **Audio**, bind your Audio Sink field targeting `sunshine_sink`.*

Create an automated display matching script inside your guest at `/usr/local/bin/sunshine-resolution.sh`:
```bash
#!/bin/bash
WIDTH=\${SUNSHINE_CLIENT_WIDTH:-1920}
HEIGHT=\${SUNSHINE_CLIENT_HEIGHT:-1080}
FPS=\${SUNSHINE_CLIENT_FPS:-60}

if [ -n "\$DISPLAY" ]; then
    MODE_STR=\((cvt\)WIDTH \(HEIGHT\)FPS | grep -v '#')
    MODE_NAME="sunshine_\${WIDTH}x\({HEIGHT}_\){FPS}"
    xrandr --newmode "\(MODE_NAME" \)MODE_STR
    xrandr --addmode Virtual-1 "\$MODE_NAME"
    xrandr --output Virtual-1 --mode "\$MODE_NAME" --primary
else
    wlr-randr --output Virtual-1 --custom-mode \${WIDTH}x\({HEIGHT}@\){FPS}Hz
fi
```
```bash
sudo chmod +x /usr/local/bin/sunshine-resolution.sh
```
*Link this file as your **Do Command** under Sunshine's Application Configuration page. Set `xrandr --output Virtual-1 --mode 1920x1080` as your **Undo Command**.*

### 3.2 Physical Client USB Redirection via Tailscale Mesh
To plug any physical USB device (like a hardware token, webcam, or security key) into your laptop and map it directly to your headless Proxmox VM workspace:

1. **On your Client Laptop (Acting as USB/IP server over Tailscale):**
   * Windows: Run `usbipd list` to locate the target device BusID, then bind it using: `usbipd bind --busid <BusID>`.
   * Linux: Run `sudo usbipd -D` followed by `sudo usbip bind -b <BusID>`.
2. **Inside your Headless Proxmox Guest VM (Claiming the device):**
   ```bash
   sudo modprobe vhci-hcd
   # Replace 100.x.y.z with your laptop's private Tailscale IP
   sudo usbip attach -r 100.x.y.z -b <BusID>
   ```

### 3.3 DeepSeek Local Execution Tuning
Deploy the native acceleration engine inside your chosen target GPU guest environment:
```bash
curl -fsSL https://ollama.com | sh
```
*   **For the RTX A2000 Slice VM:** Run `ollama run deepseek-r1:7b` to fit inside the tight VRAM boundaries.
*   **For the RTX A5000 Slice/Passthrough VM:** Run `ollama run deepseek-r1:14b` to maximize the compute capabilities of the wider memory buffer.

## Automated Client Connection Script
To make launching the showcase completely frictionless for your friend on client sites, here is an automation script to run on their laptop. Save this file as connect_workstation.sh:

#!/bin/bash
# --- CONFIGURATION CONFIGS ---
GUEST_TAILSCALE_IP="100.X.Y.Z"     # Replace with your Proxmox Guest VM's private Tailscale IP
LAPTOP_USB_BUS_ID="1-4"            # Replace with the physical BusID of your laptop's USB device
STREAM_MODE="moonlight"            # Set to "rustdesk" or "moonlight" depending on the client demo task

echo "=== [1/2] BINDING LOCAL USB DEVICE PERIPHERALS ==="if command -v usbip >/dev/null 2>&1; then
    sudo usbipd -D >/dev/null 2>&1 || true
    if sudo usbip list -l | grep -q "busid $LAPTOP_USB_BUS_ID"; then
        sudo usbip bind -b $LAPTOP_USB_BUS_ID
        echo "USB Port $LAPTOP_USB_BUS_ID isolated and ready for remote pickup."
    fifi

echo "=== [2/2] ESTABLISHING LOW-LATENCY DISPLAY STREAM ==="if [ "$STREAM_MODE" = "moonlight" ]; then
    # Connects natively to the session and matches the laptop's display configurations
    moonlight stream $GUEST_TAILSCALE_IP "Desktop" --resolution 2560x1600 --fps 90 --bitrate 35000 &else
    rustdesk --connect v23-workstation &fi

(Make the script executable on their laptop by running chmod +x connect_workstation.sh inside the terminal).
