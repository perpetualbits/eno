**Status:** working draft v0.1
**Project:** Epsilon Null Operation (ε₀)
**Companion to:** `spine_audio_dialect.md`, `spine_core_v0_3_design.md`,
`spine_runtime_model.md`, `eno_project_index.md`,

## 0. What this document is

CARVE is the offline authoring tool of ε₀. Its job is to turn raw
material — recorded instrument samples, hand-drawn 3D spaces — into
the SPINE-dialect entities that NERVE consumes at runtime. CARVE is
where the expensive, human-in-the-loop work happens; the runtime
ships only resolved data.

CARVE is not a runtime, not a parallel format, and not a DAW. It is a
node-graph editor with a coefficient-fitting backend, an IR baker,
and a SPINE-text emitter. It writes the audio dialect's canonical
entities (`audio.trajectory_template`, `audio.segment`,
`audio.impulse_response`, etc.) in their SPINE textual form. The
SPINE build pipeline turns text into the binary stream NERVE loads.

This document is re-entrant. If you come back to it in a month, §0,
§1, §2, and §11 are sufficient to recover where CARVE sits. The
remaining sections specify what CARVE produces (§1), how it talks to
sibling subsystems (§2), its architecture (§3), its three working
surfaces — 1D audio fitting (§4), 3D scene authoring (§5), IR
baking (§6) — its node-graph UI (§7), the phase plan (§8), risks
(§9), open questions (§10), and a one-page reminder (§11).

## 1. Scope and non-goals

CARVE has a deliberately narrow scope. The ε₀ project has many tools
and many subsystems; CARVE is exactly one of them and should not
absorb work that belongs elsewhere.

### 1.1 What CARVE produces

CARVE emits **SPINE text** in the audio dialect (and, in time, in
the future graphics dialect). The entities it produces are exactly
the dialect-defined types — no parallel format, no "almost-SPINE"
intermediate. Specifically:

| Entity type                          | Source of data                |
|--------------------------------------|-------------------------------|
| `audio.basis.*`                      | Dialect-fixed; CARVE just references |
| `audio.segment`                      | ML fit + hand-tune in CARVE   |
| `audio.trajectory_template`          | Assembled from segments in CARVE |
| `audio.param_spec`                   | Declared in CARVE             |
| `audio.coefficient_modulator.*`      | Declared in CARVE             |
| `audio.space.point_cloud`            | Authored in CARVE             |
| `audio.listener_grid`                | Generated by IR baker in CARVE |
| `audio.impulse_response`             | Generated by IR baker in CARVE |
| `audio.microphone_cluster`           | Authored in CARVE or default  |

CARVE may emit one entity, a small bundle, or a whole library file.
The output is always SPINE text. Binary encoding is the SPINE build
pipeline's job, not CARVE's.

### 1.2 What CARVE does NOT produce

- **SPINE atoms** directly. CARVE works at the dialect-entity level,
  not the SPINE-core operation level. It does not hand-craft DEFs,
  MODs, or LNKs except as the emission machinery for entities.
- **Score-side content** — no `music`, `cello`, or future
  motion/text dialect entities. CARVE writes the *instrument side*
  (trajectory templates, spaces, effects), never the score that
  references them. Score authoring is a separate tool (eventually) or
  is hand-written.
- **Binary SPINE format**. The build pipeline encodes; CARVE only
  emits text.
- **Runtime code**. NERVE renders; CARVE never runs at performance
  time.
- **General audio production**. CARVE is not a DAW. It does not
  mix, master, multitrack, edit MIDI, or sequence songs.

### 1.3 Non-goals

The temptation in a node-graph tool is to creep toward "general
audio workstation" or "general 3D authoring." CARVE explicitly
refuses that scope. The constraint is what makes the tool small and
the workflow tractable.

- No live performance mode.
- No real-time multi-instrument session.
- No general MIDI / OSC / DAW integration.
- No DAW-style automation lanes (expressive parameters are
  declared, not animated in CARVE; animation is the score's job).
- No general 3D modeling (`audio.space.point_cloud` only in v0.1;
  meshes/SDFs reserved for later).

## 2. Cooperation with sibling subsystems

CARVE sits between recordings/scenes and the runtime. The
relationships are explicit:

```
                                       ┌─────────────────────────┐
recordings (.wav / .flac) ─────────────│                         │
                                       │                         │
                                       │      C A R V E          │
hand-authored 3D scenes ───────────────│  (offline, x86/RV64)    │─┐
                                       │                         │ │
optional: external ML models ──────────│  ML fitting (Python)    │ │
                                       └─────────────────────────┘ │
                                                                   │ SPINE text
                                                                   │
                                       ┌─────────────────────────┐ │
                                       │  SPINE build pipeline   │◄┘
                                       │  (offline)              │
                                       │  - reachability prune   │
                                       │  - dictionary build     │
                                       │  - quantization         │
                                       │  - entropy code         │
                                       └────────────┬────────────┘
                                                    │
                                                    │ binary SPINE stream
                                                    ▼
                                       ┌─────────────────────────┐
                                       │       N E R V E         │
                                       │  (runtime, RISC-V)      │
                                       │  - load binary          │
                                       │  - schedule events      │
                                       │  - render audio/video   │
                                       └─────────────────────────┘
```

### 2.1 SPINE core

CARVE produces SPINE text following `spine_core_v0_3_design.md` §7
syntax: DEF/USE/MOD/LNK/GRP statements with the value types from §4.4.
CARVE uses *only* the dialect-defined entity types — it does not
invent its own entity classes. Cross-references between entities
(an `audio.trajectory_template` referring to its segments) use SPINE
`ref()` value syntax.

### 2.2 Audio dialect

`spine_audio_dialect.md` is CARVE's primary contract. Every entity
type CARVE produces is specified there. When the dialect changes,
CARVE changes; when CARVE needs a new entity shape, the dialect spec
gets updated first. CARVE does not silently extend the dialect.

A CARVE project records the audio-dialect *version* it was authored
against (§3.3.3). Loading a project authored against an older dialect
version is permitted; the user is warned and may re-emit.

### 2.3 NERVE (runtime)

NERVE is the downstream consumer of everything CARVE eventually emits.
CARVE never speaks to NERVE directly. The handoff is the SPINE text
file → SPINE build pipeline → binary stream → NERVE.

In early phases (before NERVE is implemented), CARVE has a small
internal playback path so the user can hear the trajectory templates
they are authoring. This is a stand-in, not a runtime: it loads a
single segment or trajectory template, applies stitch parameters,
and plays it through ALSA. The path will be removed (or repurposed
as a thin "preview" mode that delegates to NERVE) once NERVE can
render trajectory templates end-to-end. See §8 Phase 4 / Phase 6.

### 2.4 Build pipeline

The SPINE build pipeline — distinct from CARVE — does:

- reachability pruning,
- dictionary construction,
- parameter quantization (per dialect's quantization tables),
- entropy coding.

CARVE *may* perform reachability pruning at emit time (a "pruned
emit" mode, §3.4) but the build pipeline does it authoritatively
either way. CARVE's pruning is a convenience for distributing
smaller hand-off libraries.

### 2.5 Future graphics dialect

`audio.space.*` types share semantics with the future graphics
dialect: a 3D scene used to compute reverb IRs is the same scene a
shader would raycast. CARVE's 3D scene authoring path (§5) is built
with this cross-domain reuse in mind. The intent: one
`audio.space.point_cloud` is authored once and consumed by both
audio (IR baking) and graphics (rendering).

When the graphics dialect arrives, CARVE will add a graphics plugin
(§3.2) that emits its scene-aware entities; the underlying scene
authoring nodes are shared.

### 2.6 Wavelet library

CARVE depends on a wavelet/transform library that is not yet a named
subsystem in this project. CARVE assumes the library provides:

- forward and inverse transforms in each `audio.basis.*` family,
- coefficient-space arithmetic (linear ops, projection),
- (eventually) RVV-accelerated kernels.

The library will be initiated as its own subsystem in a separate
chat, with its own design document. CARVE does not specify it; CARVE
just calls into it. When the library subsystem comes online, this
section gets a named cross-reference.

## 3. Architecture

### 3.1 Two-tier implementation

CARVE is one tool but two implementation tiers:

**Tier 1 — Portable C core.** Node-graph runtime, node-graph editor
UI, coefficient editor, 3D scene editor, IR baker, project I/O,
SPINE text emitter, dialect-plugin loader. Implemented in portable
C with SDL2 + a small immediate-mode UI (microui / nuklear / hand-
rolled — chosen at Phase 2). Targets Linux x86_64 first, Linux RV64
(K3 / Jupiter / Mars) at Phase 6. No dependency on Python or any
GPU.

**Tier 2 — Python ML fitting subsystem.** Sparse coefficient fitting
of segments against wavelet bases. Runs only on the GPU development
servers (2× 32-core EPYC + 64-core EPYC; RTX A2000 + A5000). Uses
PyTorch or JAX (decision deferred to Phase 1 based on what runs best
on the RTX A2000 / A5000 under Linux). Tier 2 has no UI of its own
and is invoked from the C tier by file-based job handoff.

Communication between tiers is **file-based**, not live IPC:

```
Tier 1 (C) writes:   /tmp/carve_fit_<jobid>.in.json     (sample + basis spec)
Tier 1 invokes:      python -m carve_fit /tmp/carve_fit_<jobid>.in.json
Tier 2 (Python):     reads input, fits, writes output
Tier 1 reads:        /tmp/carve_fit_<jobid>.out.json    (coefficient list)
```

This lets Tier 1 run on a workstation while Tier 2 lives on a server
without networking glue. The schema for the job files is a CARVE-
internal interface, separate from SPINE.

**Rationale.** Tier 1 must be tiny and run on RISC-V eventually. ML
fitting is heavy, GPU-bound, and lives most naturally in Python.
Forcing all of CARVE to be portable C would either kill the fitting
work or balloon Tier 1; forcing fitting into the C editor would
prevent the work from happening on the right hardware. Separation
keeps both tiers honest.

### 3.2 Dialect-pluggable node-graph core

CARVE's node-graph core is **dialect-agnostic**. It knows about:

- typed nodes with typed ports,
- edges as port-to-port connections (using the same shape system as
  SPINE LNK: signal / value / event),
- evaluation order, propagation, dirtying,
- the immediate-mode UI for editing graphs.

It does **not** know about audio segments, point clouds, listener
grids, or wavelet coefficients. Those are contributed by **dialect
plugins**:

- `carve_audio` plugin — node types for the audio dialect.
- `carve_3d_scene` plugin — node types for shared 3D scene
  authoring (used by audio plugin in v0.1, by future graphics
  plugin later).
- (future) `carve_graphics` plugin — node types for visual
  rendering, shaders, geometry atoms.

Each plugin registers its node types, port shapes, evaluation
callbacks, and SPINE-text emitters with the core. Adding a new
dialect later means writing a plugin, not modifying the core.

Phase 1 builds the audio plugin and the 3D scene plugin together
(they are tightly coupled in v0.1: the IR baker is an audio-side
consumer of 3D-scene data). The plugin boundary becomes more
load-bearing in Phase 7 onward.

### 3.3 Native project format

CARVE has its own project file format (extension: `.carve`),
distinct from SPINE text emission.

#### 3.3.1 Round-trip

A `.carve` file persists:

- node-graph topology (which nodes, how connected),
- per-node parameter state (e.g., the user's chosen basis for each
  segment, the listener-grid resolution chosen for a scene),
- sample file references (paths, not embedded data),
- fitted coefficient data (so re-loading a project doesn't re-fit),
- hand-tuning state per coefficient,
- 3D scene contents,
- emission preferences (reachability mode, output paths).

Opening a `.carve` reconstructs the editor state exactly. Re-emitting
SPINE text from the same `.carve` produces identical (or
near-identical, modulo timestamp metadata) output.

#### 3.3.2 Format choice

JSON. Open standard, widely tooled, human-readable in a pinch,
amenable to schema validation. Schema lives in the CARVE source as
`carve_project_schema.json`. Binary or s-expression formats were
considered and rejected on tooling grounds.

(There is a parallel question for SPINE text — whether SPINE should
adopt JSON or a similar open standard rather than its current
bespoke text format. That is out of scope here; flagged for the
SPINE design chat.)

#### 3.3.3 Versioning

A CARVE project records:

- the CARVE version it was authored against,
- the audio-dialect version each audio-emitting node was authored
  against,
- (eventually) the graphics-dialect version,
- a project-format schema version.

On load, CARVE checks each version against the current build. If
the dialect version has moved, the user is shown a "may need
re-fitting" warning per affected entity; the project still loads.

### 3.4 Reachability emission modes

CARVE supports two emission modes (per the 2026-05-18 decision):

- **Pruned mode (default):** CARVE emits only the entities reachable
  from a declared "demo root" reference set. The user marks which
  trajectory templates / spaces / listener grids are the public
  entry points; everything not reachable from them is omitted.
- **Flat mode:** CARVE emits every entity it knows about, with no
  pruning. Useful for distributing an instrument library that
  downstream demos will pick from.

Both modes are first-class. The default is pruned because tiny
binaries are the project goal; flat is opt-in.

The SPINE build pipeline performs reachability pruning
authoritatively regardless of CARVE's mode. CARVE's mode just
affects what gets handed off.

## 4. The 1D audio fitting pipeline

The fitting pipeline is the workflow that produces an
`audio.trajectory_template`. It runs in five stages: sample import,
segmentation, ML fitting, hand-tuning, and template assembly.

### 4.1 Pipeline overview

```
[ Sample file ]
      │
      ▼
[ Sample import ]                 — Tier 1 (C)
      │  · WAV / FLAC read
      │  · normalize
      │  · resample if needed
      ▼
[ Segmentation ]                  — Tier 1 (C)
      │  · auto onset/body/release detection
      │  · manual override
      │  · role tagging
      ▼
[ ML fitting per segment ]        — Tier 2 (Python)
      │  · basis selection
      │  · sparse coefficient fit
      │  · quantization-aware
      ▼
[ Hand-tuning ]                   — Tier 1 (C)
      │  · coefficient-square editor
      │  · A/B between fit and edit
      │  · in-CARVE playback preview
      ▼
[ Template assembly ]             — Tier 1 (C)
      │  · order segments
      │  · set stitch_in / stitch_out
      │  · declare expressive params
      │  · emit SPINE text
      ▼
[ audio.trajectory_template ]     — SPINE text output
```

Each stage corresponds to one or more node types in the graph
(§7.2). The user can rerun any stage without restarting the
workflow: changing a basis at the fitting node re-fits and
propagates through hand-tuning and template assembly.

### 4.2 Sample import

**Inputs:** WAV or FLAC file. Sample rate can be anything libsndfile
reads; CARVE resamples to 48 kHz internally (matching the dialect's
implicit sample rate).

**Outputs:** an in-memory normalized 48 kHz mono signal, plus
metadata (original sample rate, channel count, source path).

**Operations:**

- Read with libsndfile.
- Resample to 48 kHz with a good polyphase filter (libsamplerate or
  equivalent).
- Normalize to peak amplitude 1.0.
- For multi-channel inputs: downmix to mono (sum then normalize) by
  default; user can override to pick one channel.

Stereo and multi-channel instrument samples are out of scope for
Phase 1; the trajectory template fitting model is mono, with spatial
positioning happening downstream via the listener / mic cluster
system.

### 4.3 Segmentation

**Inputs:** a normalized signal.

**Outputs:** an ordered list of segments, each tagged with a role
(`onset` / `body` / `release` / `transition` / `ornament`) and given
a duration.

**Algorithm:**

1. Compute short-time energy and spectral flux (small STFT, default
   ~10 ms window).
2. Detect onsets via spectral flux peaks above a threshold.
3. Detect release boundaries via energy decay below a threshold for
   N consecutive frames.
4. Anything between onset and release is body.
5. Compute initial role tags from these boundaries.
6. Present the result in the UI for manual override.

Manual override is mandatory in practice: most real instrument
recordings have ambiguous segment boundaries (a slow bow attack
blurs onset and body; a vibrato body has internal events that look
like onsets). The auto-segmenter's purpose is to save time on the
clear cases, not to be authoritative.

The user can:

- adjust segment boundaries by dragging,
- merge two segments,
- split one segment,
- re-tag a segment's role,
- delete a segment (e.g., trimming a long silence at the end).

### 4.4 ML fitting (Tier 2)

For each segment, the fitting node runs:

1. **Basis selection.** Either fixed by the user (when they know
   what they want) or chosen by a small per-segment heuristic
   (default mapping: onset → Morlet; body → chirplet; release →
   damped_exp; transition / ornament → user-selected).
2. **Sparse fit** of the segment against the selected basis,
   producing a coefficient list in the dialect's expected shape per
   §3.2 of `spine_audio_dialect.md`.
3. **Quantization-aware optimization** (when budget permits): fit
   directly in the quantized parameter space, not in float space
   followed by quantization. This avoids coefficients drifting after
   the optimizer thinks it's done.

**Fitting algorithms per basis** (sketches; details deferred to a
future `carve_ml_fitting.md`):

| Basis           | Algorithm sketch                                  |
|-----------------|---------------------------------------------------|
| `morlet`        | Matching pursuit on a Morlet dictionary; sparse LASSO refinement. |
| `chirplet`      | Chirplet decomposition (Mann-Haykin) + nonlinear refinement on chirp rate. |
| `gabor`         | Like Morlet but real-valued; simpler dictionary. |
| `damped_exp`    | ESPRIT or Prony for modal estimation. |
| `formant_stack` | Formant tracker (LPC roots) + Gaussian envelope per formant. |
| `noise_fricative` | Bandpass parameter fit on log-spectrum envelope; phase irrelevant. |
| `impulse`       | Time-localized peak detection; one coefficient per peak. |

**Sparsity target:** ~5 bytes per wavelet after quantization (per
the audio dialect §3.2). Per-segment target: 8–40 wavelets.

**Output:** a coefficient list in the dialect's per-basis shape, plus
a residual signal (the part of the input the basis didn't capture).
The residual is visualized but not stored; its size relative to the
input is the canonical "fit quality" metric in the UI.

### 4.5 Hand-tuning

The fitted coefficients are an estimate. The user almost always
wants to refine them — to push perceptually load-bearing
coefficients, suppress fitting artifacts in the tail, or match the
fit to a creative intent the algorithm doesn't share.

The primary surface is the **coefficient square** (§7.2.2):

- The bottom row is the coarsest scale (one cell, covering the whole
  segment).
- Each row above doubles the number of cells, halving their time
  width.
- Two opposing hues encode positive vs negative coefficient sign.
- Luminance encodes coefficient magnitude.
- A reconstructed waveform is overlaid (faint) for orientation.

The user clicks/drags cells to scale or zero them, or selects a
region to mute it. The waveform overlay updates in real time. A
play button auditions the current state through the in-CARVE
playback path (§4.6).

For bases with named coefficients (chirplet's chirp rate; formant
stack's formant frequencies; etc.) the coefficient square is
supplemented with named-parameter sliders on a side panel. These
are the perceptually load-bearing handles.

**Three view modes** (per §7.2.2):

- **Static**: one square, one timeslice. The default for hand-tuning
  a single segment.
- **Timeline**: a row of squares spanning a longer signal, side by
  side. Used for inspecting segmentation and basis choice across an
  entire instrument note.
- **Scrolling**: one square stays put, the signal scrolls through
  it. Used for live preview while playing.

**I/Q dual-square mode**: for periodic signals (most of the body
segments), CARVE represents the signal as an (I, Q) pair. Phase
progression in the original signal becomes a rotation in the I/Q
plane — i.e., a `R(φ) · (I, Q)` matrix multiplication. The dual
square shows the I and Q coefficient fields side by side. This makes
phase-dependent effects (vibrato, tremolo, phase modulation) easy to
read and tune: a smooth circular trace in I/Q corresponds to a clean
tone, deviations correspond to noise or modulation.

### 4.6 In-CARVE playback (preview path)

For Phases 1–4 (until NERVE renders trajectory templates end-to-
end), CARVE has its own playback path:

- Takes a single segment or an assembled trajectory template.
- Walks the basis's inverse transform on the coefficients.
- Applies stitch_in / stitch_out crossfades between segments.
- Routes through ALSA at 48 kHz.

This is a stand-in. Once NERVE can render trajectory templates,
CARVE's playback path is either removed entirely (the user previews
through NERVE) or repurposed as a thin client that hands templates
to NERVE for live preview. The crossover point is decided at Phase 6
based on NERVE's state.

### 4.7 Expressive parameter declaration

After fitting, the user declares which expressive parameters the
trajectory template exposes to the score, and how each modulates
the underlying coefficients. This produces `audio.param_spec` and
`audio.coefficient_modulator.*` entities.

In v0.1, only two modulator types are wired:

- `audio.coefficient_modulator.linear_scale` — scales the
  amplitude of a coefficient axis. (E.g., `vibrato_depth` scales an
  inner-amplitude coefficient.)
- `audio.coefficient_modulator.linear_shift` — shifts a coefficient
  along a frequency or time axis. (E.g., `vibrato_rate` shifts an
  inner-frequency coefficient.)

`crossfade_pair` (between two pre-fit segment variants) is reserved
for Phase 3, once the second variant fits are in the workflow.

The user authors these in the ParamSpecAuthor node (§7.2.1). The
declared parameters propagate to the trajectory template's
`expressive_params` field.

### 4.8 Trajectory template assembly

The final stage. The user:

- orders the segments,
- sets stitch_in / stitch_out durations between adjacent segments,
- sets the template's `pitch_reference` (the natural pitch the
  template was fit at) and `duration_reference`,
- lists the expressive parameters from §4.7.

CARVE emits the resulting `audio.trajectory_template` as SPINE text,
along with the segments and any new modulators / param specs. The
text is human-readable and matches the schema in
`spine_audio_dialect.md` §3.3.

## 5. 3D scene authoring (shared with graphics)

### 5.1 The shared substrate

Per the 2026-05-17 decision: SDF, raycasting, and polar wavelet
reverb share the same substrate — a 3D scene plus a query point plus
radial accumulation. CARVE authors the scene once; both audio (via
the IR baker, §6) and future graphics (via a future shader pipeline)
consume the same scene definition.

In v0.1, "the scene" is an `audio.space.point_cloud`. Higher-
fidelity scene types (`audio.space.mesh`, `audio.space.sdf`) are
reserved in the audio dialect's type catalog but not implemented in
CARVE yet.

### 5.2 Point cloud authoring (v0.1)

**Node types:** `PointCloudEditor`, `ListenerPlacement`,
`MicrophoneClusterPlacement`.

**Workflow:**

- Drop points in 3D space (mouse drag in a 3D view, or numeric
  entry).
- Each point has position (x, y, z) and reflectivity ∈ [0, 1].
- Position the listener inside the cloud.
- Position the microphone cluster relative to the listener (or use
  one of the dialect's default clusters).
- The scene is visualized as wireframe with points as small markers
  and the listener as a small camera glyph.

**Output:** an `audio.space.point_cloud` entity + an `audio.listener`
+ optionally a custom `audio.microphone_cluster`.

### 5.3 Future scene types

`audio.space.mesh` and `audio.space.sdf` are reserved type ids in
the audio dialect. When they arrive in CARVE:

- Mesh: imported from OBJ/PLY or authored from primitives.
- SDF: composed from a small algebra of distance functions (sphere,
  box, plane, union, intersection, smin) — the demoscene-standard
  SDF authoring style.

Both are deferred. The Phase 4 IR baker is built against point
clouds only; the abstraction is kept loose enough to accept other
scene types later without rewriting the baker's interface.

### 5.4 Cross-domain note

Until a graphics dialect exists, CARVE emits `audio.space.*` only.
When the graphics dialect arrives, the same scene entity is
referenced from a (future) `graphics.scene` or similar; CARVE's
output gains a sibling entity that lives in the graphics dialect
namespace but shares the underlying scene data.

## 6. IR baker (polar wavelet reverb)

The IR baker is the offline machinery that turns a 3D scene + a
listener grid + a microphone cluster into a set of impulse
responses (`audio.impulse_response` entities), bundled into an
`audio.listener_grid`.

### 6.1 What it bakes

**Inputs:**

- An `audio.space.*` entity (point cloud in v0.1).
- An `audio.listener` (defines the listener's center pose, the
  microphone cluster, and — for grids — the region of expected
  listener motion).
- A listener-grid specification (a set of grid points where IRs are
  baked).

**Outputs:**

- One `audio.impulse_response` per (grid point, microphone) pair.
- One `audio.listener_grid` bundling the IRs with their positions.

### 6.2 Algorithm sketch (Approach 3)

For each grid point, for each microphone in the cluster:

1. **Cast radial rays** from the listener (= grid point) outward.
   Number and angular density of rays are baker parameters.
2. **Intersect rays with reflectors.** For point clouds:
   ray-point proximity within a small spatial tolerance counts as a
   hit, weighted by the point's reflectivity.
3. **Bounce up to N times** (N is a baker parameter, default 2–3).
   Each bounce accumulates time delay (distance traveled / speed of
   sound) and amplitude (cumulative reflectivity, attenuation per
   distance squared).
4. **Frequency shaping per bounce** (Phase 4+): per-reflector
   absorption coefficients in two or three frequency bands. v0.1
   uses a single broadband reflectivity scalar.
5. **Accumulate taps** as (time_offset, amplitude) pairs per
   microphone.
6. **Radial / temporal bucketing** to keep the IR sparse: collapse
   taps within a small time window into a single representative tap.
   Target: 30–100 nonzero taps per IR.
7. **Quantize** to the dialect schema (`i14` time offset, `ulog.10`
   amplitude per `audio.impulse_response` §3.7 of
   `spine_audio_dialect.md`).

The result is a sparse FIR per (grid point, microphone). NERVE runs
the convolution at runtime; CARVE just produces the taps.

### 6.3 Pre-echoes

Per the decision on 2026-05-17: pre-echoes are taps with negative
time offsets relative to the dry signal but within the global audio
latency window L (typically 100–300 ms). NERVE introduces L at
playback; the baker is allowed to emit negative-time taps freely
within [-L, 0).

For v0.1, negative-time taps are author-introduced (the user can
explicitly request a pre-echo as a marker on the scene authoring
side) or are a side-effect of acausal post-processing on the baked
IR. They will not arise from the basic ray-bounce algorithm above,
which is causal by construction.

### 6.4 Grid resolution heuristic

Listener grid resolution is a trade-off:

- Dense grid → smooth listener motion, more storage.
- Sparse grid → smaller storage, audible artifacts on rapid motion.

CARVE provides a sensible default and an explicit override:

- **Default:** ~0.5 m grid spacing within the scene's bounds, on a
  rectilinear grid clipped to the listener's expected motion region
  (an author-declared bounding box; defaults to the scene's bounding
  box if not specified).
- **Override:** user sets a custom grid (any density, any shape) via
  the ListenerGrid node. Non-rectilinear grids are allowed (e.g., a
  sphere of points around a key listener position).

This resolves `spine_audio_dialect.md` §12 open question #3 at the
CARVE level by making it user-controllable with a useful default.

### 6.5 Interpolation contract

CARVE emits the grid + IRs in their absolute form. NERVE
interpolates between neighboring grid IRs at runtime using
trilinear (rectilinear grid) or barycentric (irregular grid)
weights. The decision that this interpolation is valid lives in the
2026-05-17 decision log entry; the rationale (wavelet transform
linearity) is recorded there.

CARVE does **not** pre-interpolate — that would defeat the purpose
of storing only grid IRs.

### 6.6 Memory accounting

For sizing intuition:

- 1 IR ≈ 30–100 taps × ~3 bytes per tap (after dialect quantization)
  ≈ 100–300 bytes.
- 1 stereo grid IR ≈ 2 × IR ≈ 200–600 bytes.
- 1 small grid (4 × 4 × 2 = 32 grid points) × 600 bytes ≈ 20 KiB.

A 4k demo can afford one IR (the listener doesn't move, so grid
collapses to one point), one stereo, ~600 bytes. A 64k demo can
afford a real grid for a moving listener. CARVE reports the memory
budget in the IR baker node's status panel after each bake.

## 7. Node-graph UI

CARVE's user-facing surface is a node-graph editor in the style of
patchers (Pure Data, Max, TouchDesigner) but constrained to its
narrow scope. This section sketches the architecture and the v0.1
node catalog. Detailed UX polish is deferred (§8 Phase 6).

### 7.1 Architecture

The core node-graph runtime (§3.2) provides:

- A typed graph: nodes with named typed ports.
- Edges connect output ports to input ports of matching shape.
- Evaluation order is determined by topological sort + dirty
  propagation: when a node's input changes, downstream nodes are
  marked dirty and re-evaluated on the next pass.
- Re-evaluation is bounded: expensive operations (ML fitting, IR
  baking) are explicit "run" buttons rather than auto-triggered.
  A coefficient hand-tune doesn't re-run fitting; it only re-emits.
- Evaluation runs on the main thread; long jobs (Tier-2 fitting,
  IR baking) run on a worker thread / subprocess and report back via
  the same dirty mechanism.

The UI shell is a single window with three panes:

- **Graph pane (left):** node-graph canvas; pan/zoom; drag to wire.
- **Parameter pane (right):** the selected node's parameters.
- **Preview pane (bottom):** context-dependent — the coefficient
  square for fitting nodes, the 3D scene viewport for scene nodes,
  the IR sparkline for the IR baker.

Keyboard-first navigation; mouse for graph wiring and 3D
manipulation. Visual style per §7.3.

### 7.2 Key node types (v0.1)

The audio plugin contributes the audio-side nodes; the 3D scene
plugin contributes the spatial nodes.

#### 7.2.1 Audio plugin nodes

| Node                  | Inputs                | Outputs                  | Purpose                                |
|-----------------------|-----------------------|--------------------------|----------------------------------------|
| `SampleSource`        | file path (param)     | normalized signal        | §4.2: WAV/FLAC load, normalize         |
| `Segmenter`           | signal                | segment list             | §4.3: onset/body/release split, role tags |
| `BasisSelector`       | segment               | segment + basis tag      | §4.4: per-segment basis choice         |
| `SparseFitter`        | tagged segment        | fitted segment (coeffs)  | §4.4: Tier-2 ML fit; long-running button |
| `CoefficientEditor`   | fitted segment        | hand-tuned segment       | §4.5: coefficient-square hand-tune     |
| `ParamSpecAuthor`     | (segments)            | param_spec entities      | §4.7: declare expressive parameters    |
| `TrajectoryAssembler` | segments + param specs| trajectory_template      | §4.8: order, stitch, emit              |
| `Preview`             | segment or template   | audio out (ALSA)         | §4.6: in-CARVE playback                |
| `SpineEmitter`        | any entity nodes      | SPINE text file          | §3.4: write the .spine file, pruned or flat |

#### 7.2.2 The coefficient square

The hand-tuning surface (§4.5) is its own visual primitive,
implemented in the audio plugin (with the 3D scene plugin
inheriting a similar visual treatment for spatial coefficient
fields in the future).

A coefficient square is a square or near-square grid:

- **Vertical axis: wavelet levels**, labelled `level · count_per_level`
  (e.g. `0·128`, `1·64`, … `7·1` for a 128-sample 7-level
  decomposition; `0·2048`, `1·1024`, … `12·1` for a 4096-sample
  12-level decomposition).
- **Horizontal axis: time** within the represented slice. Top row =
  finest time resolution, many narrow cells. Bottom row = coarsest, one
  cell spanning the whole window.
- **Cell color:** two opposing hues encode coefficient sign;
  luminance encodes magnitude. The exact hue pair is taken from the
  cyberpunk palette (§7.3): teal for one sign, magenta-pink for the
  other.
- **Overlay (optional):** the reconstructed signal drawn faintly over
  the grid, for orientation.

**Three view modes:**

- **Static.** One square showing one slice. Default for hand-tuning
  a single segment.
- **Timeline.** A row of squares laid side by side, covering a
  longer signal. Used for inspecting segmentation and basis choice
  across an entire instrument note. Each square represents a fixed-
  duration slice (e.g., 85.3 ms at 48 kHz / 4096 samples).
- **Scrolling.** A single square stays in place; the signal scrolls
  through it. Used for live preview while playing.

**I/Q dual-square mode** for periodic signals (most body segments):
two coefficient squares side by side, one for the I coefficient
field, one for the Q coefficient field. A clean tone shows a smooth
circular trace in the I/Q plane; deviations show as noise or
modulation in one or both squares. Useful for tuning vibrato,
tremolo, and phase modulation, where a phase progression in the
signal is a rotation in (I, Q).

The coefficient square is load-bearing: it's how the user *sees* a
wavelet-domain signal. Sliders supplement it for named coefficients,
but the square is the primary surface.

#### 7.2.3 3D scene plugin nodes

| Node                  | Inputs                | Outputs                  | Purpose                                |
|-----------------------|-----------------------|--------------------------|----------------------------------------|
| `PointCloudEditor`    | (param: points)       | point_cloud entity       | §5.2: 3D point placement               |
| `ListenerPlacement`   | (param: pose)         | listener entity          | §5.2: listener pose                    |
| `MicrophoneClusterPlacement` | (param: cluster) | microphone_cluster entity | §5.2: per-channel mic offsets         |
| `ListenerGrid`        | listener + scene bounds | listener_grid spec     | §6.4: grid layout                      |
| `IRBaker`             | scene + grid + cluster | listener_grid + IRs     | §6.1–6.2: bake IRs; long-running button |

### 7.3 Cyberpunk visual style (notes)

A few concrete commitments now; UX polish later. The intent is a
dense, technical, slightly unfriendly look — the editor is a tool
for a person who already knows what they're doing.

**Palette:**

- Base: very dark, near-black with a faint cool cast
  (e.g. `#0a0a12`).
- Foreground text: light grey (e.g. `#e0e0e8`), not pure white.
- Accent (active state, selection, positive coefficient):
  cyan-teal (e.g. `#33ffe6`).
- Accent (warning, negative coefficient, error):
  magenta-pink (e.g. `#ff337a`).
- Secondary accent (info, dim glyphs): muted purple-grey.

**Typography:**

- Monospaced for all numeric and structural UI (port labels, node
  parameters, coefficient values, timecode). IBM Plex Mono or
  similar open-source monospace.
- Variable-width sans for descriptive labels and tooltips.
- Tabular figures everywhere (no jumping columns).

**Motifs:**

- Subtle scanline or pixel-grid backgrounds (low contrast, easy to
  ignore).
- Sharp 1-pixel borders. No drop shadows, no rounded corners over
  a few pixels, no gradients in chrome.
- Wireframe 3D rendering for scenes (not shaded).
- Coefficient squares use the teal/magenta accent pair as their sign
  encoding, tying the visual identity to the work surface.

**Anti-goals:**

- No skeuomorphism (no virtual cables, no knobs that imitate
  hardware).
- No animation longer than ~150 ms. UI must feel snappy on RV64.
- No splash screen, no chrome wasting more than 32 pixels of edge.

The full design is deferred to a Phase 6 polish pass. Until then,
the editor uses these baselines and looks plain-but-consistent.

## 8. Phase plan

The phase plan trades early breadth for one working slice end-to-
end as fast as possible. Each phase produces a concrete artifact
that the next phase builds on.

### Phase 1 — One-shot fit-and-emit (CLI, no UI)

**Goal:** prove the data path. No editor yet.

**Work:**

- Tier 2 Python: load a WAV (one cello D4 body), fit it against the
  chirplet basis with whatever algorithm comes up first (matching
  pursuit or chirplet decomposition).
- Tier 2: emit a JSON description of the fit (the Tier-1↔Tier-2
  interchange).
- Tier 1 C: stub CLI tool that reads the JSON, wraps it as a
  hand-written `audio.segment` + `audio.trajectory_template` SPINE
  text file.
- Tier 1 C: minimal playback CLI tool (`carve-preview`) that reads
  a SPINE text file, walks the chirplet inverse transform, and plays
  through ALSA.

**Artifact:** `cello_d4_body.spine` — a SPINE text file containing
one segment in one trajectory template. The file plays audibly via
`carve-preview`. Decision point: which ML framework (PyTorch vs JAX)
based on what worked.

**Out of scope:** UI, project file, multiple segments, hand-tuning,
3D scenes, IR baking.

### Phase 2 — Minimal C node-graph editor

**Goal:** establish the editor shell.

**Work:**

- Choose immediate-mode UI library (nuklear / microui / hand-rolled).
- Implement the node-graph core (§3.2): typed nodes, typed ports,
  topological evaluation, dirty propagation.
- Implement the three-pane window layout (§7.1).
- Implement two nodes: `SampleSource`, `SpineEmitter`.
- Implement `.carve` project file load/save (JSON, §3.3).
- Connect the cyberpunk palette and monospace typography at
  baseline level (no polish).

**Artifact:** opens a `.carve` project file, loads a sample,
displays it in the preview pane, emits a stub SPINE text file
containing a placeholder segment. The editor runs on Linux x86_64.

**Out of scope:** fitting, hand-tuning, 3D scenes.

### Phase 3 — Audio fitting integrated as nodes

**Goal:** the full 1D pipeline (§4) inside the editor.

**Work:**

- Wire Tier 1 → Tier 2 file-based job invocation (§3.1).
- Add nodes: `Segmenter`, `BasisSelector`, `SparseFitter`,
  `CoefficientEditor`, `ParamSpecAuthor`, `TrajectoryAssembler`,
  `Preview`.
- Implement the coefficient square (§7.2.2) in all three view modes,
  with I/Q dual-square for chirplet / Gabor / Morlet bases.
- Implement the in-CARVE preview path (ALSA playback of a
  trajectory template, §4.6).
- Implement `crossfade_pair` modulator if Phase 1 hasn't already
  needed it.

**Artifact:** load a cello note recording, run the full pipeline
end-to-end, emit a complete `audio.trajectory_template` SPINE text
file, audition through the preview node. One full cello-body fit
from sample to SPINE text takes one editing session, not a week.

**Out of scope:** 3D scenes, IR baking.

### Phase 4 — 3D scene + IR baker

**Goal:** spatial audio offline workflow.

**Work:**

- Add the 3D scene plugin (§3.2): nodes per §7.2.3.
- Implement the 3D wireframe viewport in the preview pane.
- Implement the IR baker (§6) for point cloud scenes, with the
  grid-resolution heuristic (§6.4).
- Wire the baker's output into `SpineEmitter` so a complete project
  (instrument library + scene + baked IRs) emits as one file.
- Listener grid resolution UX: defer detailed control until this
  phase has baked at least one real scene.

**Artifact:** a SPINE text file containing the cello D4 trajectory
template plus a small point-cloud room and a stereo listener with a
baked IR. The preview path applies the IR by sparse convolution and
plays the result.

**Out of scope:** RISC-V port, expressive parameter coverage beyond
the basics, second instrument.

### Phase 5 — Full instrument coverage

**Goal:** cover one instrument end-to-end with realistic
expressiveness.

**Work:**

- Fit all four standard cello segments (onset, body, release,
  optional transition) for the first cello note.
- Assemble the cello.gesture.legato trajectory template.
- Wire up the expressive parameters that the cello dialect actually
  uses: bow_pressure, vibrato_depth, vibrato_rate, brightness.
- Cross-check against the cello dialect's gesture catalog (in
  `spine_dialect_template.md` §3) to make sure CARVE's output is
  consumable.

**Artifact:** a SPINE library file containing a usable cello note
with full gesture support. The file plugs into a hand-written cello
score and renders through the preview path (or NERVE, if NERVE has
reached this point).

**Out of scope:** RISC-V port.

### Phase 6 — RISC-V port + polish

**Goal:** the editor runs on RISC-V hardware; visual style is
polished.

**Work:**

- Port Tier-1 C to RISC-V (K3 / Jupiter / Mars). Tier-2 stays on
  x86/GPU; the C client connects to a remote Tier 2 over the same
  file-based interface (with SSH/SFTP file movement).
- First polish pass of the cyberpunk visual style (§7.3).
- Decide playback path: continue with in-CARVE ALSA playback, or
  delegate to NERVE if NERVE is far enough along (§4.6).
- Make `crossfade_pair` modulator work on real fit data.

**Artifact:** CARVE running on a K3, editing the cello project from
Phase 5, with the cyberpunk style applied. ML fitting still happens
on the x86 server; everything else is local.

**Out of scope:** second instrument (Phase 7).

### Phase 7 — Second instrument + graphics handshake

**Goal:** prove generality. Confront the first cross-instrument
question.

**Work:**

- Fit a second instrument. Provisional pick: violin (similar to
  cello so shared infrastructure can be tested) or a wind instrument
  (different enough to surface assumptions). Decision deferred.
- Surface and resolve any cross-instrument shared-basis questions
  in the project format and emission.
- Implement a placeholder graphics plugin that consumes the same
  3D scene entity and produces a stub graphics-dialect output. This
  is a thin handshake, not real rendering.

**Artifact:** a SPINE library containing two instruments and one
shared scene, with both audio and graphics consumers wired into the
scene. Documents the cross-instrument shared-basis question with
evidence from real fits.

## 9. Risks and mitigations

| Risk                                                       | Mitigation                                                  |
|------------------------------------------------------------|-------------------------------------------------------------|
| ML fitting quality varies wildly across bases.             | Handle bases independently. Hand-tuning is mandatory, not optional. The coefficient square makes residual error visible. |
| Coefficient quantization error compounds across segments.  | Fit in quantized space when possible, not float-then-quantize. Compare quantized-fit playback to original sample at hand-tuning time. |
| Node-graph UI in C from scratch is a large undertaking.    | Use a small immediate-mode UI library (microui / nuklear). UI is functional, not polished, until Phase 6. No live styling system. |
| IR baker quality on point-cloud scenes may be poor.        | Allow explicit hand-placed early reflections in addition to ray-cast diffuse tail. Listen-and-adjust workflow. |
| Tier-1 / Tier-2 boundary causes friction.                  | File-based handoff with stable JSON schema. No live IPC. Tier 1 can run with stale fits when Tier 2 is unavailable. |
| Audio-side decisions leak into 3D scene UX.                | Dialect plugins are explicit from Phase 2. Audio and 3D share the node-graph core but contribute distinct node families. |
| The audio dialect evolves and breaks old `.carve` files.   | Per-node dialect version field (§3.3.3). Old projects load; user is warned when re-emit is needed. |
| ML framework choice locks in heavy dependencies.           | Phase 1 picks based on what works on RTX A2000/A5000 under Linux; switch is cheap because fitting is the only consumer and the interchange is JSON. |
| Coefficient-square rendering performance on RISC-V.        | The display is a 2D color grid plus a faint polyline; trivial in GPU terms. If software rendering matters on early RV64 boards, fall back to coarser cell granularity. |
| Wavelet library subsystem doesn't exist yet.               | CARVE specifies the interface it needs; the library subsystem is initiated in its own chat. CARVE can stub a single-basis path for Phase 1. |

## 10. Open questions

1. **ML framework choice.** PyTorch or JAX or something else?
   Decided in Phase 1 by what actually works on the A2000/A5000
   under Linux.
2. **Listener grid resolution UX.** Per-room user-set or auto from
   scene bounds with override? Defer to Phase 4 when there's a real
   scene to bake.
3. **Cross-instrument shared bases.** When two instruments share a
   basis family, do their fitting nodes share data, or each fit
   independently? Defer to Phase 7 when there are two instruments.
4. **In-CARVE playback vs delegate to NERVE.** Default position:
   small in-CARVE path through Phase 4; delegate to NERVE once
   NERVE can render trajectory templates. Final cutover decided at
   Phase 6.
5. **Native project format internal layout.** JSON is the format;
   the exact schema (how the graph topology is encoded, how
   embedded coefficient data is stored, whether sample files are
   embedded or referenced) is settled in Phase 2.
6. **SPINE text format itself.** Should SPINE adopt JSON or another
   open standard rather than its bespoke text format? Out of scope
   here; flagged for the SPINE design chat.
7. **Coefficient-square cell selection UX.** Drag-select rectangles,
   marquee, click-individual? Settled in Phase 3 once real fits are
   visible.
8. **Wavelet library subsystem name and design.** CARVE depends on
   this library but does not specify it. The library is initiated
   in its own chat with its own design document. When that subsystem
   exists, §2.6 here gets a named cross-reference.


That is CARVE.
