# SMOLA: A Python-Preprocessed Macro Language for RISC-V Assembly

## Status

Design document v1. No implementation yet. Companion to SMOLR and smold.

## Project identity

**SMOLA** is a thin, line-oriented preprocessor that turns `.smola` source
files into GAS-compatible `.s` files, which standard `as`/`ld` then turn
into RISC-V object files and executables. SMOLA lives at `tools/smola/`
inside the ENO monorepo.

SMOLA is *not* an assembler. It does not encode instructions, generate
relocations, or emit ELF. It does exactly one thing: it expands a small
set of high-level directives (named registers, struct field access,
function frames, method calls) into ordinary RISC-V assembly text that
GAS can consume.

The motivating problem is that GAS macros are not a real programming
language — no data structures, no scopes, no string parsing — and so any
attempt to add OOP-style affordances inside `.macro` blocks ends up
fighting the tool. SMOLA moves the macro layer into Python, where data
structures and error reporting are normal.

## 1. Design principles

### 1.1 Zero runtime cost

Every SMOLA construct expands to the same machine instructions that a
careful assembly programmer would have written by hand. SMOLA decides
*which physical register a name binds to* and *what offset a field
lives at*, but the resulting `.s` file contains nothing the programmer
could not have written directly. If a SMOLA-generated function is
larger than a hand-written equivalent, the macro layer is wrong and
must be fixed.

### 1.2 Provenance is preserved

Every emitted instruction is annotated with the source-level intent
that produced it. `add s2, s3, t0` is fine; `add s2, s3, t0  # total
= subtotal + tax` is what SMOLA emits. The `.s` file is meant to be
readable when something goes wrong. This is not a compiler — debugging
happens by reading the generated assembly, and the generated assembly
must reward that reading.

### 1.3 Line-oriented, not expression-oriented

One source line is one directive or one pseudo-instruction or one
pass-through line. No nested expressions, no infix syntax, no operator
precedence rules. This keeps the parser trivial and the mental model
identical to ordinary assembly. The user is writing assembly; SMOLA is
spelling fewer letters for them.

### 1.4 Escape hatch is always available

Any line beginning with `!` passes through to the output unchanged
(with the `!` stripped). This means SMOLA never has to be complete —
when the macro layer is missing a feature, the user drops to raw
assembly for one line. The escape hatch is not a workaround; it is
part of the design.

### 1.5 Generated output is a build artifact, not source

`.s` files generated by SMOLA are checked into `build/`, not into
version-controlled source directories. They are reproducible from the
`.smola` source and the SMOLA version. They are inspected during
debugging but never edited by hand.

### 1.6 No round-tripping

SMOLA is a one-way translation from `.smola` to `.s`. There is no plan
to disassemble `.s` (or worse, a linked binary) back to `.smola`. That
would be a separate, much harder project — and one that the smold tool
might one day cover from a different direction. SMOLA stays one-way.

## 2. The language

### 2.1 Lexical structure

A SMOLA source file is a sequence of UTF-8 lines. Each line falls into
one of five categories, decided by its first non-whitespace character
and the syntactic shape that follows:

| First token             | Line type                |
|-------------------------|--------------------------|
| `!`                     | Raw pass-through         |
| `#` or `//`             | Comment (also passed through, with translation to `#`) |
| `.smola.<keyword>`      | SMOLA directive          |
| `<name>:`               | Label (pass-through)     |
| `.<gas-directive>`      | GAS directive (pass-through, but may carry SMOLA hints inline) |
| anything else           | Instruction or pseudo-instruction line |

Whitespace is not significant beyond separating tokens. Tabs and
multiple spaces are equivalent.

Identifiers follow C rules: `[A-Za-z_][A-Za-z0-9_]*`. Struct fields and
method names may use `.` as a path separator (`Point.x`, `Point.translate`).

### 2.2 Comments

`#` and `//` both introduce comments. SMOLA passes them through to the
output as `#` comments (the GAS native form). End-of-line comments after
instructions are preserved verbatim, after register-name substitution if
the line is a SMOLA pseudo-instruction.

### 2.3 SMOLA directives

All SMOLA directives begin with `.smola.` to distinguish them from
GAS directives. The full set in v1 is small. Future versions may add
more, but each addition must be justified by code it actually shortens
in real ENO/SMOLR/smold sources.

#### `.smola.func <name>`

Begin a function definition. Establishes a new scope: a fresh register
allocator state, an empty saved-register set, and an empty named-binding
table. Emits the GAS section header and label per the smold section-per-
function convention.

Expands to:

```
    .section .text.<name>, "ax", @progbits
    .globl  <name>
    .type   <name>, @function
    .balign 2
<name>:
```

Optionally, `.smola.func <name> static` omits the `.globl` directive.

#### `.smola.endfunc`

End a function definition. This is the directive that does real work:

1. Walks the body emitted since `.smola.func` and discovers which
   callee-saved registers were claimed via `VAR.S`.
2. Generates a prologue (stack adjustment + saves) and prepends it to
   the function body.
3. Generates an epilogue (restores + stack adjustment + `ret`) and
   appends it.
4. Emits the `.size <name>, .-<name>` directive.
5. Releases all named bindings.

This two-pass behavior is the central reason SMOLA exists. GAS cannot
do it; Python can, trivially.

#### `.smola.struct <Name> { <fields> }`

Declare a struct layout. Fields are comma-separated, each with a name
and a type:

```
.smola.struct Point {
    x: i64,
    y: i64,
}
```

Supported field types in v1:

| Type    | Size (bytes) | Alignment | GAS load/store |
|---------|--------------|-----------|----------------|
| `i8`    | 1            | 1         | `lb` / `sb`    |
| `u8`    | 1            | 1         | `lbu` / `sb`   |
| `i16`   | 2            | 2         | `lh` / `sh`    |
| `u16`   | 2            | 2         | `lhu` / `sh`   |
| `i32`   | 4            | 4         | `lw` / `sw`    |
| `u32`   | 4            | 4         | `lwu` / `sw`   |
| `i64`   | 8            | 8         | `ld` / `sd`    |
| `u64`   | 8            | 8         | `ld` / `sd`    |
| `ptr`   | 8            | 8         | `ld` / `sd`    |
| `<S>`   | sizeof(S)    | align(S)  | (inlined struct, no direct load/store) |

Field offsets are computed at preprocess time using simple
size-and-align rules: each field is placed at the next position
divisible by its alignment. Total struct size is rounded up to the
struct's alignment (= max of field alignments). No bitfields. No
explicit padding directives in v1.

The struct definition emits `.set Point_x_offset, 0` style symbols to
GAS for reference in passthrough lines and for debuggers. It also
records the layout in the SMOLA preprocessor's symbol table so that
later `LOAD_FIELD` / `STORE_FIELD` directives can resolve.

#### `.smola.method <Struct>.<name>`

Sugar for `.smola.func <Struct>_<name>`. The first argument register
(`a0`) is automatically bound to the name `self` for the duration of
the function. Equivalent to:

```
.smola.func <Struct>_<name>
    VAR.A self                          # bound to a0
```

#### `.smola.endmethod`

Sugar for `.smola.endfunc` paired with `.smola.method`. Identical
semantics; named differently for symmetry.

### 2.4 Register-binding pseudo-instructions

These are the workhorses. They look like instructions but expand to
zero machine code (they only update SMOLA's internal state) or to
exactly one comment line.

#### `VAR.T <name>` — claim a caller-saved (temporary) register

Allocates one of `t0`..`t6` (in that order), binds the given name to
it, and emits a comment line:

```
    # smola: bind <name> -> t<N>  (caller-saved)
```

Caller-saved registers are claimed without affecting the prologue.
They are released at the next `FREE <name>` or implicitly at
`.smola.endfunc`. If the pool is exhausted, SMOLA reports an error
identifying which names currently hold which registers, so the user can
decide whether to free something or restructure.

#### `VAR.S <name>` — claim a callee-saved register

Allocates one of `s0`..`s11` (in that order). Records the choice so
that `.smola.endfunc` knows to save and restore it. Emits:

```
    # smola: bind <name> -> s<N>  (callee-saved, frame slot reserved)
```

#### `VAR.A <name>` — bind to a fixed argument register

`VAR.A <name>` without an explicit register binds to the next free
argument register starting at `a0`. `VAR.A <name> a3` binds explicitly
to `a3`. Argument registers do not need to be released; their lifetime
is the function. SMOLA tracks them only to detect collisions (binding
two names to `a0` is an error).

#### `VAR.RET <name>`

Sugar for `VAR.A <name> a0`. Documents intent at the call site of a
return value.

#### `FREE <name>`

Release a name. Its physical register returns to the free pool. Using
a name after `FREE` is an error. `FREE` on an argument or return-value
binding is a no-op with a warning (you cannot free `a0` mid-function
without breaking the ABI).

#### `VAR.ALIAS <new_name> = <existing_name>`

Create a second name for the same physical register. Useful when a
value is conceptually two things at once (a count *and* a loop index).
`FREE` on either name releases the register; the other becomes dangling
and any use is an error.

### 2.5 Pseudo-instructions

Every regular RISC-V instruction is also accepted as a SMOLA
pseudo-instruction if all of its register operands are SMOLA names.
SMOLA looks up each name, substitutes the physical register, and emits
the resulting line.

```
    ADD result, a, b               # -> add s2, s3, t0
    LD value, 0(self)              # -> ld s4, 0(a0)
    BEQZ count, .Ldone             # -> beqz t1, .Ldone
```

Mixing names and explicit registers is allowed but discouraged:

```
    ADD result, a, t6              # -> add s2, s3, t6   (t6 is raw)
```

Capitalization rule: instructions in **all uppercase** are SMOLA
pseudo-instructions (operand substitution happens). Instructions in
**lowercase** pass through unchanged. This lets you reach raw assembly
in one line without the `!` prefix when convenient. Both styles can
coexist in one function.

### 2.6 Field access

Struct field access takes two forms.

#### Direct: `LOAD_FIELD dst, base, Struct.field`

Expands to the appropriate load instruction at the computed offset:

```
LOAD_FIELD x, self, Point.x        # -> ld s2, 0(a0)
LOAD_FIELD age, person, Person.age # -> lhu t0, 16(s5)
```

`STORE_FIELD src, base, Struct.field` is the matching store.

#### Address-of: `LA_FIELD dst, base, Struct.field`

Computes the address of a field without loading it:

```
LA_FIELD p, self, Point.coords     # -> addi t0, a0, 8
```

(Uses `addi` when the offset fits in 12 bits, which is the normal case
for structs. Larger offsets emit `addi` + `add` or fall back to
`auipc` / `addi` if needed. The expander chooses; the user does not.)

### 2.7 Method calls

`CALL Struct.method, arg1, arg2, ...`

Arranges arguments in `a0`..`a7` (the first argument is `self`,
implicitly bound at the callee side by `.smola.method`), emits the
`call Struct_method` instruction, and on return restores any
caller-saved bindings that were live across the call.

In v1, "arranges arguments" means emitting `mv` instructions for each
argument that is not already in its target register. SMOLA does not
yet do parallel-move resolution (handling cycles like `a0 <- a1, a1 <-
a0` correctly). v1 detects such cycles and reports an error. v2 may
add a real parallel-move pass.

A bare `CALL <symbol>, arg1, ...` (no `Struct.` prefix) is allowed and
just emits a regular `call`.

### 2.8 What v1 does NOT have

To keep the implementation honest and the surface small:

- **No virtual dispatch.** Method calls are always static, resolved
  at preprocess time. See SMOLR §11.1 for why.
- **No inheritance.** A struct cannot extend another struct in v1.
  (The layout convention "derived struct begins with its base's fields"
  is achievable by hand using nested struct fields.)
- **No generics or templates.** If you want a vector of `i32` and a
  vector of `i64`, write them as two structs.
- **No scope-tracked destructors.** Manual `FREE` only.
- **No conditional assembly.** Use GAS `.if`/`.endif` as pass-through.
- **No `include`.** All sources are flat for now. v2 may add it.
- **No expression evaluation.** Constants in field offsets and
  immediates are passed through to GAS, which already handles them.

Each of these is a candidate for v2 once v1 has been used in anger.

## 3. Worked example

A SMOLA source file:

```
.smola.struct Point {
    x: i64,
    y: i64,
}

.smola.method Point.translate
    VAR.A dx                       # bound to a1
    VAR.A dy                       # bound to a2
    VAR.T cx                       # bound to t0
    VAR.T cy                       # bound to t1

    LOAD_FIELD cx, self, Point.x
    LOAD_FIELD cy, self, Point.y
    ADD cx, cx, dx
    ADD cy, cy, dy
    STORE_FIELD cx, self, Point.x
    STORE_FIELD cy, self, Point.y
.smola.endmethod
```

The generated `.s` file:

```
    .set Point_x_offset, 0
    .set Point_y_offset, 8
    .set Point_size, 16
    .set Point_align, 8

    .section .text.Point_translate, "ax", @progbits
    .globl  Point_translate
    .type   Point_translate, @function
    .balign 2
Point_translate:
    # smola: bind self -> a0  (argument)
    # smola: bind dx -> a1  (argument)
    # smola: bind dy -> a2  (argument)
    # smola: bind cx -> t0  (caller-saved)
    # smola: bind cy -> t1  (caller-saved)
    ld   t0, 0(a0)                 # LOAD_FIELD cx, self, Point.x
    ld   t1, 8(a0)                 # LOAD_FIELD cy, self, Point.y
    add  t0, t0, a1                # ADD cx, cx, dx
    add  t1, t1, a2                # ADD cy, cy, dy
    sd   t0, 0(a0)                 # STORE_FIELD cx, self, Point.x
    sd   t1, 8(a0)                 # STORE_FIELD cy, self, Point.y
    ret
    .size Point_translate, .-Point_translate
```

No callee-saved registers were claimed, so no prologue or epilogue
overhead beyond `ret`. Size: 6 instructions plus `ret`. Identical to
what a careful hand-author would write.

## 4. Architecture

### 4.1 Pipeline

```
.smola source
      ↓
   tokenize (line by line)
      ↓
   parse into a sequence of typed events:
      DirectiveEvent / BindingEvent / InstructionEvent / PassthroughEvent
      ↓
   walk events maintaining:
      - global symbol table (structs, methods)
      - per-function state (register allocator, frame plan, body buffer)
      ↓
   emit .s file with provenance comments
```

### 4.2 Modules

```
tools/smola/
    smola/
        __init__.py
        lexer.py            # line → tokens
        parser.py           # tokens → events
        symbols.py          # struct table, method table
        regalloc.py         # register allocator: pools, bindings, scopes
        emitter.py          # event → .s line(s), with provenance
        frame.py            # prologue/epilogue planner
        errors.py           # SmolaError with source location
    bin/
        smola               # CLI entry: smola input.smola -o output.s
    tests/
        test_lexer.py
        test_parser.py
        test_regalloc.py
        test_struct.py
        test_function.py
        test_end_to_end.py
        fixtures/
            *.smola
            *.s.expected
    docs/
        Smola_Spec.md       # this document
```

### 4.3 The register allocator

The allocator state per function:

```python
class Allocator:
    free_T: list      # available t0..t6
    free_S: list      # available s0..s11
    free_A: list      # available a0..a7 not yet bound
    bindings: dict    # name -> Binding(reg, kind, source_line)
    saved_S: set      # which s-regs need prologue saving
```

Allocation order is deterministic and documented. v1 uses round-robin
(lowest-numbered free register first) for predictability. v2 may add
a preference hint for RVC compressibility (prefer registers in the
RVC subset: `x8`–`x15`, i.e. `s0`, `s1`, `a0`–`a5`).

Errors caught by the allocator:

- pool exhausted (with full current state in the message)
- double-bind (same name twice without intervening `FREE`)
- use-after-free (name referenced after `FREE`)
- argument-register collision (two `VAR.A` to the same physical reg)
- bind to reserved register (`zero`, `ra`, `sp`, `gp`, `tp`) is rejected
  unless the user explicitly opts in with `VAR.RESERVED`

### 4.4 The frame planner

The frame planner runs at `.smola.endfunc` time. Inputs:

- the set of `s`-registers claimed via `VAR.S`
- whether `ra` is live across any call in the body (always assumed
  true in v1; v2 may track this and omit `ra` save for leaf functions)
- any user-requested stack space (`.smola.stack <N>` directive,
  declared inside the function body — v1 includes this for raw
  spills, even though we expect not to need it often)

Output:

- prologue: `addi sp, sp, -frame_size` + `sd ra, ...(sp)` + `sd
  s0..., ...(sp)`
- epilogue: matching loads + `addi sp, sp, frame_size` + `ret`
- the prologue is *prepended* to the buffered function body, not
  emitted at `.smola.func` time. This is the critical design point.

Frame size is rounded up to 16 (RISC-V calling convention requires
16-byte stack alignment at call sites).

Leaf-function optimization (no calls in body, no `s`-registers
claimed): no prologue, no epilogue, just `ret`. v1 implements this
because it is trivial and common.

### 4.5 Error reporting

Every error carries:

- the source file name
- the source line number (1-based)
- the source line text
- a caret marker if the error has a column
- the current allocator state if relevant (e.g. on pool exhaustion)

Errors halt processing. SMOLA does not try to recover and continue.
The user fixes the first error and re-runs.

### 4.6 Determinism

Given the same input file and the same SMOLA version, the output `.s`
file is byte-identical. No timestamps, no random ordering, no hash-based
register choice. This is required for reproducible builds and for
sensible diffs when sources change.

## 5. CLI

```
smola [options] input.smola

Options:
    -o, --output <file>     Output .s file. Default: input with .smola
                            replaced by .s.
    --stdout                Emit to stdout instead of a file.
    --no-provenance         Suppress # smola: ... comments. Smaller
                            output, harder to debug. Off by default.
    --tier <0|1|2>          Reserved for future use (RVC register
                            preference). v1 ignores this.
    --check                 Parse and check only; do not emit output.
    -v, --verbose           Print progress on stderr.
```

Exit codes: 0 success, 1 SmolaError, 2 internal error / bug.

## 6. Build integration

A `make` rule:

```
%.s: %.smola
	$(SMOLA) $< -o $@

%.o: %.s
	$(AS) $(ASFLAGS) $< -o $@
```

`.smola` files are versioned. `.s` files are built artifacts and live
in `build/`. The SMOLA version is recorded in the `.s` header as a
comment, so a `.s` file checked in for debugging can be associated
with the SMOLA that produced it.

## 7. Testing strategy

SMOLA is tested at four levels:

1. **Unit tests** for the lexer, parser, register allocator, and
   frame planner. Pure Python, fast.
2. **Golden-file tests**: a corpus of `.smola` fixtures, each with an
   expected `.s` output. Mismatch fails the test and prints a diff.
3. **Assembly tests**: each generated `.s` must assemble cleanly with
   `riscv64-linux-gnu-as`. CI runs this.
4. **Behavioral tests**: a small number of SMOLA-written programs run
   under `qemu-riscv64` and must produce the expected output. v1
   target: a SMOLA port of `smold_insn_length` from smold M1, which
   must produce a bit-identical object file to the hand-written
   version (modulo provenance comments).

The fourth test is the load-bearing one: it proves that SMOLA is
zero-cost in practice, not just in theory.

## 8. Milestones

### M0: Design

This document.

### M1: Lexer, parser, smallest useful runtime

Tokenizer, line-typing parser, register allocator with VAR.T/VAR.A/FREE,
function frames without VAR.S (leaf functions only), basic pseudo-
instruction substitution, pass-through, escape hatch, provenance
comments. End-to-end: a SMOLA file with one leaf function compiles to
working `.s` that assembles. Golden-file tests for ~10 small fixtures.

### M2: Structs and field access

Struct declarations, `LOAD_FIELD`, `STORE_FIELD`, `LA_FIELD`. Method
declarations with implicit `self`. Golden-file tests for the Point
example.

### M3: Callee-saved registers and prologue/epilogue

VAR.S, frame planner, the two-pass body-then-prologue emission. This
is the first milestone where the macro layer does work no manual
discipline can do. Behavioral test: port a non-leaf smold atom and
verify the linked object is byte-identical to the hand-written one
(after stripping provenance comments).

### M4: Method calls and argument shuffling

`CALL Struct.method, args...`. v1 simple-move semantics (no parallel
move). Cycle detection in the move resolver. Caller-saved-live-across-
call tracking, so SMOLA can spill if needed (v1: error if a caller-
saved binding is needed across a call; the user must free or spill
manually).

### M5: Port smold M1 atoms

Rewrite `tools/smold/src/core.S` (or a representative subset) in
`.smola`. Verify size, behavior, and that the disassembled output
matches. This is the proof that SMOLA is useful, not just designed.

### M6: First SMOLR use

When SMOLR's runtime resolver assembly stabilizes (Phase 3 in the
SMOLR plan), rewrite it in SMOLA. This is the second proof of value.

### M7: Polish

Better error messages, optional RVC register preference hint, optional
parallel-move resolver, `include`, conditional assembly via SMOLA
rather than GAS pass-through.

## 9. Risks

### Risk: SMOLA-generated assembly is larger than hand-written

Mitigation: golden-file tests compare bit-for-bit against hand-written
references. Any size regression is a bug to fix, not a feature to
accept.

### Risk: error messages are unhelpful

Mitigation: every error carries source location and relevant allocator
state. Real-world use will surface bad error cases; treat them as
bugs.

### Risk: feature creep makes SMOLA into a compiler

Mitigation: the §2.8 non-goals list is enforced. Every proposed v2
feature must demonstrate code it shortens in real SMOLR or smold
sources, not hypothetical ones.

### Risk: SMOLA hides bugs that would have been obvious in raw assembly

Mitigation: provenance comments. Every emitted instruction shows the
source line that produced it. The `--no-provenance` flag exists but
is off by default. Reading the generated `.s` must remain the natural
debugging path.

### Risk: register allocator picks pessimal registers for RVC

Mitigation: v1 uses deterministic round-robin. v2 adds an RVC
preference hint. Until then, golden-file tests on real smold atoms
will reveal whether the v1 policy is actually worse than hand
allocation.

### Risk: two-pass frame planning desyncs from body emission

Mitigation: the function body is buffered as a list of fully-formed
output lines plus a metadata header. The frame planner only prepends;
it never rewrites body lines. This isolates the failure mode.

## 10. Relationship to SMOLR and smold

- SMOLA is **not** a SMOLR dependency. SMOLR works fine with raw
  GAS assembly; SMOLA is a productivity layer for whoever writes that
  assembly.
- SMOLA-generated `.s` files participate in SMOLR's pipeline like any
  other `.s` file. SMOLR sees only the assembled object, which has
  no SMOLA-specific structure.
- smold disassembles SMOLA-generated binaries the same way it
  disassembles any other RISC-V binary. There is no special tooling
  for "SMOLA-aware disassembly" — that would be the round-tripping
  trap from §1.6.
- The §1.2 provenance comments in the `.s` file are *not* preserved
  in the object file (GAS strips them). If smold ever wants to map
  back to SMOLA source, that would be a separate metadata channel
  (DWARF, or a SMOLA-specific side file). Not a v1 concern.

## 11. Final note

SMOLA exists to remove friction, not to add abstraction. If the
generated `.s` becomes harder to read than the hand-written version
it replaces, something has gone wrong. The user is still writing
assembly. SMOLA just types less of it on their behalf.

## 12. Implementation notes (v0.1 retrospective)

Items below were clarified during the v0.1 implementation and belong
in the spec, not in commit messages.

### 12.1 Local labels are section-scoped

GAS local labels (`.L<name>`) are scoped to the section they appear
in. Because `.smola.func` opens a new `.text.<name>` section per
function, a `.Lloop` defined inside function `A` cannot be referenced
from function `B`. This matches smold's section-per-function
discipline and is the desired behavior; it is documented here so that
new users do not try to use local labels as cross-function jump
targets. For cross-function jumps, use a global symbol declared in
the target function.

### 12.2 Bare identifiers in pseudo-instructions must resolve

In a SMOLA pseudo-instruction (uppercase mnemonic), every operand
that is a bare identifier must be either:

- a name previously bound by `VAR.T`/`VAR.S`/`VAR.A`/`VAR.RET`/`VAR.ALIAS`
- a raw register name (`t0`, `a1`, `x10`, etc.)
- a local label (starts with `.`, e.g. `.Lloop`)
- a numeric immediate

If none of these apply, SMOLA raises a `RegAllocError` with a hint to
declare the name or to use the `!` escape hatch. This is intentional:
silently passing through an unknown bare identifier as a "label"
hides the most common typo class. To branch to or call a global
symbol from a SMOLA pseudo-instruction context, use the `!` escape
hatch for that line, or call it through `CALL <symbol>, ...` which
handles globals explicitly.

### 12.3 `FREE` semantics for `s` registers

`FREE` on a `VAR.S` binding removes the name from the binding table
but does not return the physical register to the free pool. The
register stays committed to the prologue save list for the rest of
the function. This is intentional: returning an `s` register to the
pool would allow a later `VAR.S` to alias the same physical register
under a different name, which is confusing to read in the generated
output for a vanishingly small size benefit. `FREE` on an `s` binding
is therefore documentation-only.

`FREE` on `VAR.T` does return the register to the pool, as expected,
and is the primary mechanism for register reuse in tight functions.

`FREE` on argument bindings (`VAR.A` / `VAR.RET`) is also documentation-
only, because the calling convention fixes their physical meaning.

### 12.4 Two-pass body emission

The translator buffers all body lines for a function and only emits
them at `.smola.endfunc` time, prepended with the prologue computed
from the body's `VAR.S` claims and call detection. This is the load-
bearing reason SMOLA exists rather than living as GAS macros. The
buffer is a list of fully-formed output strings; the planner never
rewrites a body line, only prepends.

This also means errors emitted from inside a function body still
carry the original source location, because the location is captured
at lex time and travels with the event, not with the output text.

### 12.5 Multi-line `.smola.struct` is folded pre-lex

A `.smola.struct Foo { ... }` block written across multiple lines is
folded onto a single line before lexing. This is a workaround for the
fact that lines like `    x: i64,` would otherwise lex as labels.
The fold preserves source line numbers by emitting blank placeholder
lines for the consumed continuations, so error messages for code
*after* a multi-line struct still point at the right line.
